# tag: i18n — Internationalisierung (beide Locales)

> **Wer trägt diesen Tag:** frontend, website, content (alles, was übersetzbare Inhalte berührt).
> Querschnitts-Tag — die Review-Rolle prüft i18n-Parität bei jedem Tick.

## Pflichten

- **ALLES translated:** keine hardcoded Strings ohne `$t()`-Key. Jeder Key existiert in **beiden** Locales (de+en).
- **Verify auf BEIDEN Locales Pflicht** vor „done" — nicht nur Default. (Loki-Lehre Docs-Round-3.)
- **Default-Locale = `en`:** `/` = en, `/de` = de (PO-Anforderung, projekt-justierbar via Override).
- **URL ↔ Locale-Konsistenz:** non-de-URLs müssen englisch sein, wenn EN Default ist (`/contact` statt
  `/kontakt`). Page-Filenames pro Locale via i18n-`pages`-Mapping.
- **i18n-Parität testbar machen:** eine Spec, die prüft, dass jeder Key in beiden Locales nicht-leer ist
  (Regressions-Anker gegen fehlende Übersetzungen).
- **id-basierte Lookups, keine geratenen Slugs** (i18n-Slug-Mismatch war ein Production-Bug) — siehe [`api.md`](api.md).
